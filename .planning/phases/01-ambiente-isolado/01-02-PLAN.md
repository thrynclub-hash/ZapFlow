# Phase 1 Plan 02: Branch fase-2 + deployment Vercel isolado

## Objective
- **What:** Publicar a branch git `fase-2`, criar um projeto Vercel novo e dedicado (`zapflow-fase2`) conectado a essa branch, configurar as env vars `VITE_*` apontando para o Supabase isolado da Plan 01-01, e fazer o deploy inicial.
- **Why:** É o critério de sucesso mais visível da fase — Leonardo precisa conseguir abrir uma URL e ver a Fase 2 rodando, sem tocar em nada de produção. Um projeto Vercel dedicado (não apenas um Preview de branch dentro do projeto de produção) garante isolamento estrutural: nenhuma env var pode vazar entre prod e fase-2 porque são *projetos* diferentes, não apenas *environments* diferentes do mesmo projeto.
- **Output:** Branch `fase-2` publicada, projeto `zapflow-fase2` deployado e acessível numa URL própria, com env vars corretas.

## Tasks

<task type="code">
  <name>Enviar commits pendentes de main para origin antes de criar a branch</name>
  <files>git (thrynclub-hash/ZapFlow)</files>
  <action>`git rev-list --count origin/main..main` para confirmar quantos commits locais estão pendentes (identificado na pesquisa como Pitfall #9; a contagem exata muda a cada sessão — não travar num número fixo). `git push origin main`. Isso precisa acontecer ANTES de criar `fase-2`, porque o Vercel vai importar o repo do GitHub (não do disco local) e a branch `fase-2` precisa nascer de um `main` remoto atualizado.</action>
  <verify>`git log origin/main..main` não mostra diferença (0 commits à frente) após o push; `git status` mostra "up to date with origin/main".</verify>
  <done>main local e origin/main sincronizados.</done>
</task>

<task type="code">
  <name>Criar e publicar a branch fase-2</name>
  <files>git</files>
  <action>`git checkout -b fase-2 main` e `git push -u origin fase-2`.</action>
  <verify>`git ls-remote --heads origin fase-2` retorna uma ref.</verify>
  <done>origin/fase-2 existe.</done>
</task>

<task type="code">
  <name>Criar o projeto Vercel dedicado</name>
  <files>Vercel CLI</files>
  <action>`vercel project add zapflow-fase2` no escopo `leonardo-marussos-projects-8d614f58` (único escopo Vercel confirmado autenticado nesta sessão — a pesquisa não encontrou nenhum projeto "zapflow" pré-existente nele; ver nota de verificação abaixo, não bloqueante). Em seguida `vercel link --yes --project zapflow-fase2` a partir da raiz do repo (idealmente com `fase-2` já checked out).</action>
  <verify>`vercel project ls` mostra `zapflow-fase2`; `.vercel/project.json` (gitignored) referencia o projeto.</verify>
  <done>Projeto criado e linkado localmente.</done>
</task>

<task type="code">
  <name>Conectar o projeto Vercel ao repositório GitHub</name>
  <files>Vercel CLI ou dashboard</files>
  <action>`vercel git connect` para linkar `thrynclub-hash/ZapFlow` a `zapflow-fase2` — habilita deploy automático a cada push. Se o comando não suportar modo não-interativo neste ambiente, usar o dashboard (Settings → Git) como alternativa.</action>
  <verify>`vercel project inspect zapflow-fase2` mostra o repositório git linkado.</verify>
  <done>Repo GitHub conectado, auto-deploy habilitado.</done>
</task>

<task type="code">
  <name>Configurar variáveis de ambiente isoladas</name>
  <files>Vercel CLI: vercel env add</files>
  <action>Buscar `get_project_url(project_id=<fase2_ref da Plan 01-01>)` e `get_publishable_keys(project_id=<fase2_ref>)`. Para cada ambiente (Production, Preview, Development) do projeto `zapflow-fase2` (projeto dedicado, então não é necessário escopar por branch): `vercel env add VITE_SUPABASE_URL` e `vercel env add VITE_SUPABASE_ANON_KEY`, colando os valores buscados. NUNCA copiar `VITE_SUPABASE_URL`/`VITE_SUPABASE_ANON_KEY` de `.env`, `.env.example` ou `INTEGRATIONS.md` — esses descrevem produção (Pitfall #7, aplicado aqui ao invés de só ao cron). Não editar o CSP em `vercel.json` — `connect-src 'self' https://*.supabase.co wss://*.supabase.co` já cobre qualquer projeto novo via wildcard, nenhuma mudança necessária (Pitfall #8).</action>
  <verify>`vercel env ls` mostra `VITE_SUPABASE_URL` e `VITE_SUPABASE_ANON_KEY` configuradas em `zapflow-fase2`, com valores contendo a ref do projeto novo (nunca `bhiggyigsrqfabqhutne`).</verify>
  <done>Ambas as vars presentes nos ambientes relevantes, confirmadamente distintas dos valores de produção.</done>
</task>

<task type="code">
  <name>Deploy inicial</name>
  <files>Vercel CLI: vercel deploy</files>
  <action>A partir da branch `fase-2`, rodar `vercel deploy` (preview) ou `vercel deploy --prod` (para este projeto dedicado tratar como sua própria "Production" — qualquer um satisfaz o critério de sucesso, a diferença é só o rótulo/URL no dashboard).</action>
  <verify>Deploy conclui com sucesso e retorna uma URL ao vivo.</verify>
  <done>URL de deployment registrada.</done>
</task>
